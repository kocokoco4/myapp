## 五線譜SVG描画ルール（`src/utils/staff.ts`）

**情報源**: Elaine Gould "Behind Bars" (Edition Peters), Lilypond Notation Reference, SMuFL仕様 (W3C Music Notation Community Group)

### 座標系（現行実装）
- `SS = 8`（五線の線間＝1 staff space）、`SH = 4`（半スペース＝隣接位置間）
- `pos 0` = 第1線（最下線）、`pos 8` = 第5線（最上線）
- Y座標: `pY(pos, staffBot) = staffBot - pos * SH`（pos↑ = Y↓ = 画面上方向）

### A. 音部記号と音名対応
| ト音記号（treble） | ヘ音記号（bass） |
|---|---|
| 第1線 = E4 | 第1線 = G2 |
| 第2線 = G4 | 第2線 = B2 |
| 第3線 = B4（中線） | 第3線 = D3（中線） |
| 第4線 = D5 | 第4線 = F3 |
| 第5線 = F5 | 第5線 = A3 |

### B. 加線（Ledger Lines）— Gould pp.18-19
- 五線の**最初の間（space）にある音**には加線不要（例: G5はト音の上の間、加線なし）
- 加線が必要な最初の位置: ト音下 = C4（pos -2）、ト音上 = A5（pos 10）
- 加線は音符の位置まで全ての「線」位置（偶数pos）に描く
- 加線の幅: 符頭の両端から**約1/3 staff space**ずつはみ出す
- **推奨上限: 3〜4本**。それ以上は8va記号やクレフ変更を検討

### C. 符幹（Stem）— Gould p.15
- **中線（第3線）以上 → 下向き**（符頭の左側）
- **中線より下 → 上向き**（符頭の右側）
- **中線上の音（B4/D3）: 下向き**（慣例）
- 最小長: **3.5 staff space**（= 28px in current system）
- 加線の音: 符幹は**最低でも中線まで到達**させる
- 連桁グループ: グループ内で中線から最も遠い音で方向を決定。同距離なら下向き

### D. 休符の位置 — Gould p.37
| 休符 | 位置 |
|------|------|
| 全休符 | 第4線（pos 6）から**吊り下げ**。第3-4線間を占める |
| 2分休符 | 第3線（pos 4）の**上に載せる**。第3-4線間を占める |
| 4分休符 | 五線の中央約**3 staff space**にわたるジグザグ形状 |
| 8分休符 | 約1.5〜2 staff space。中線付近に配置 |
| 16分休符 | 8分と同様だが旗2つ分で約2〜2.5 staff space |

### E. 符頭（Note Head）— SMuFL/Bravura
- 幅: 約**1.2 staff space**（楕円形、幅 > 高さ）
- 傾き: 約 **-10°〜-20°**（フォントにより異なる。現行 -10°）
- 塗りつぶし: 4分音符以下 = filled、2分音符 = hollow + stem、全音符 = hollow のみ

### F. 連桁（Beaming）— Gould pp.153-155
- 8分音符以下が連桁対象。4分音符以上は連桁しない
- **4/4拍子**: 半小節（beat 3）を跨ぐ連桁は禁止。`[beat1-2][beat3-4]`
- 8分音符: 1拍単位（2音）または半小節単位（4音）でグループ化
- 16分音符: 1拍単位（4音）でグループ化
- 主連桁の太さ: 約 **0.5 staff space**
- 連桁間の隙間: 約 **0.25〜0.33 staff space**

### G. SVG高さの動的計算（実装ルール）
- 全音符のpos範囲をスキャンし、五線の上下にはみ出す分だけSVG高さを拡張する
- コードテキストは常にSVG最上部（音符より上）に配置
- シラブル（歌詞割当）がある場合、五線下にさらに16px確保

## フェーズ3 仕様（β公開・収益化）

### 3-A. 雰囲気ベース伴奏生成

#### UI設計
- 4カテゴリのプルダウンで雰囲気を選ぶ
- 選んだ結果を「○ × ○ × ○ × ○ = ？」の数式UIで表示
- 「生成」ボタンで答えが出る演出にする

#### カテゴリ（プルダウン）
1. **感情**: 泣きたい夜 / 前を向く朝 / 胸がざわざわする / ほっとする / ときめいてる / 怒ってる
2. **景色**: 雨の窓際 / 夏の終わり / 冬の帰り道 / 夜の街 / 朝の光 / 懐かしい場所
3. **テンション**: 静かに燃えてる / 叫びたい / ゆっくり歌いたい / 踊りたい
4. **関係性**: 好きな人へ / 遠くなった人へ / 自分へ / 誰かへ

#### 生成の仕組み
- Gemini APIでタイトル（例：「霧雨のセレナーデ」）とキャッチコピーを生成
- テンプレートから雰囲気に合ったBPM・リズム・コード感で伴奏を生成
- AIによる伴奏カスタマイズは別ステップ（有料機能）として分離

#### 提案判定方法
- Geminiプロンプトに「提案を含む回答の場合は冒頭に[PROPOSAL]をつけて」と指示
- フロント側で`[PROPOSAL]`を検出 → Firestoreのカウントを+1

## ZooLab 構想との連携指針（全アプリ共通）

### ZooLab とは

オーナーが開発するすべてのアプリは **ZooLab** というプラットフォームブランドに属する。
ZooLab は「動物テーマ × 使いやすさ × 中小向け価格」を差別化軸とした、自社開発アプリのマーケットプラットフォームである。

#### サービス階層

| ティア | 対象 | 内容 |
|--------|------|------|
| Zooタウン | 個人・趣味 | 身近な動物・植物モチーフの軽量アプリ |
| Zooナショナル | 小規模ビジネス | 外部連携・サテライト型（既存ツールを強化） |
| Zooワールド | 中規模企業 | 自社オリジナルアプリ（独立稼働） |
| Zooギャラクシー | エンタープライズ | 基幹・DWH・全社データ連携基盤 |

#### 設計思想

- 各アプリ = **APIを持つ独立したブロック（レゴモデル）**
- 自社アプリ同士は DB 共有または API で連携可能な設計を維持する
- 外部サービス（スマレジ・freee・Slack 等）とは API 連携で繋ぐ
- API の外部公開は行わない（世界観・品質を守るクローズドプラットフォーム）
- 将来的にすべてのアプリが **Zooギャラクシー（基幹プラン）** に統合できる構造を常に意識する

## API・外部連携 実装指針（全アプリ共通）

### 基本方針

新機能を追加するときは、**常に「他のZooLabアプリや外部サービスと連携できるか」を設計段階で考慮する。**
現時点で連携しなくてもよいが、**後から繋げられる構造**にしておくこと。

### 必須設計ルール

1. **データ構造を汎用的に保つ**
   - アプリ固有すぎるキーを避ける
   - 将来 JSON API として出力できる形式でデータを保持する

2. **外部連携ポイントには必ずコメントを残す**
   ```typescript
   // [ZooLab連携ポイント] ここで他アプリにデータを渡す想定
   // 将来: POST /api/zoolab/[app-name]/[endpoint]
   ```

3. **認証を意識した設計**
   - ユーザー識別子（ID）を持つデータは必ず設計に含める
   - 将来の JWT / OAuth 対応を想定してユーザー情報を分離して管理する

4. **外部連携候補を常にコメントで明示する**
   ```typescript
   // [外部連携候補] スマレジAPI / freee API / Slack Webhook
   // 現状はローカル処理だが、後でここを差し替える
   ```

5. **環境変数・APIキーは定数として分離する**
   - `src/constants.ts` の `CONFIG` オブジェクトに集約

### Webhook・通知連携の準備

データが更新されるタイミングには必ず以下のコメントを入れる：
```typescript
// [Webhook候補] このイベントを外部に通知できる
// 例: Slack通知 / メール送信 / 他アプリへのデータ反映
```
